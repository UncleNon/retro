# 1. 結論サマリー
結論から言うと、本案件は『Godot 4.x + Tiled + Aseprite』を中核にし、AI画像生成は『Nano Banana 2 / Nano Banana Pro』と『GPT Image 1.5』を役割分担させる二段構えが最適である。再現すべきは単なる4色・低解像度ではなく、“狭い画面でも迷わない設計”“短い周回ループ”“強い収集圧”“データの外部化による量産可能性”である。Nintendo公開情報ではオリジナルGame Boyは160 x 144画素・4階調で、Pan Docsは8x8/8x16スプライトや40スプライト/画面・10スプライト/走査線という制約を示している [R1][R2]。この制約が、シルエット優先・UI簡素化・地形再利用・高密度テキスト誘導という設計思想を生んだ。

また、Square Enixが『ドラゴンクエストモンスターズ テリーのワンダーランドRETRO』をスマートフォン向けに展開し、オリジナル準拠のピクセル表現・簡潔なシステム・UIカスタマイズを現代端末で成立させていることから、同種の表現が今でも十分に実用的であることは確認できる [R3]。一方で、同じ“雰囲気”を狙っても、固有キャラクター・地名・アイコン・メニュー枠・モンスター外形を寄せ過ぎると著作権・商標・トレードドレス上のリスクが上がるため、本企画は必ずオリジナルIPとして設計する。

> **本書で採用する最終推奨**
>
> エンジンはGodot 4.x、マップ編集はTiled、最終ドット制作はAseprite、AI素材生成はNano Banana 2/Proで高速な案出し・多バリエーション生成、GPT Image 1.5で厳密な差分編集と参照画像ベースの一貫性維持を行う。Midjourneyはムードボード用途、Adobe Fireflyは企業案件のブランド整合用途、Recraftはベクター/UI補助用途として扱い、主制作基盤にはしない。

| **項目**       | **推奨**                                    | **理由**                                                                                     |
|----------------|---------------------------------------------|----------------------------------------------------------------------------------------------|
| ゲーム構造     | トップダウン探索 + 3体PTターン制 + 合成育成 | DQM級の育成中毒性と携帯機向け短いセッション性を両立しやすい。                                |
| 画面設計       | 内部解像度 320x288 / 可視20x18マス          | Game Boyの20x18タイル密度を維持しつつ、現代端末で文字とUIを読みやすくする。                  |

| マップ運用     | Tiled JSON + custom properties              | 配置・フラグ・ワープ・ショップ・会話トリガをデータ駆動で管理しやすい [R6]。                |
| 最終ドット制作 | Aseprite                                    | tilemap layer、linked cels、CLI exportで量産/修正/自動出力に強い [R7]。                    |

# 2. 調査結果: 何を再現すべきか
当時の携帯機RPGを“それっぽく”見せていた本質は、低性能そのものではない。本質は、制約に合わせて情報設計を圧縮し、プレイヤーが一画面内で『今やるべきこと』を迷わないように整理した点にある。Nintendoの仕様ではGame Boyは160 x 144画素・4階調 [R1]、Pan Docsでは8x8/8x16 sprite、40 sprite/画面・10 sprite/lineが示されている [R2]。このため、当時の優れた作品は背景・UI・キャラを同時に主張させず、“一目で判別できる差分”だけを残した。ここを再現しない限り、単にドット化しても初代級の品質にはならない。

| **観測された事実**                                  | **画面上の結果**                         | **本企画での要件化**                                                       |
|-----------------------------------------------------|------------------------------------------|----------------------------------------------------------------------------|
| 160x144 / 20x18タイル級の狭い視界 [R1][R2]      | 常時表示情報が少ない。探索が密に感じる。 | 可視20x18マスを維持。常時HUDは最小化し、メニューに機能を寄せる。           |
| スプライト数/線数制限 [R2]                        | 同一画面の被写体数が抑制される。         | 街・ルート・ダンジョンごとに同時注目対象を3つ以内に制御する。              |
| 4階調や少色数 [R1][R2]                          | 輪郭と塊の読みやすさが最優先になる。     | 1キャラ1特徴、1タイル1役割、1UI面1主題で設計する。                         |
| 現代スマホでのRETRO移植成立 [R3]                  | 現代端末でも成立する。                   | モバイル操作は追加するが、ゲームループ自体は古典的でよい。                 |
| Godotのinteger scaling推奨 [R5]                   | 中途半端な拡大でドットが滲まない。       | 内部解像度固定 + 整数拡大を標準とする。                                    |
| Tiled custom properties / templates / worlds [R6] | データ駆動で配置・分割・再利用しやすい。 | マップ/イベント/NPC/ワープを外部データ化し、デザイナが直接編集可能にする。 |

## 2.1 再現原則
原則A: 1画面1課題。プレイヤーが『話す』『拾う』『進む』『戻る』のどれをすべきか一目で分かること。

原則B: 1キャラ1記号。髪型・角・尻尾・帽子・肩幅など、遠目でも識別できる差異を優先すること。

原則C: 地形は文法で作る。草・土・崖・水・壁の接続ルールを固定し、バリエーションは文法の範囲で増やすこと。

原則D: UIは常時主張しない。探索時は静か、戦闘とメニューでのみ情報密度を上げること。

原則E: 収集→強化→新領域解放の1サイクルを10〜20分で回せるようにすること。

原則F: ドットの精度は“解像度の低さ”ではなく“塊の整理”で決まる。ぼかしやAAに逃げないこと。

原則G: データは全部外に出す。モンスター、技能、遭遇、ショップ、イベント条件はコードに焼き込まないこと。

原則H: “昔っぽい”ではなく“今遊んでも気持ちいい”。倍速、オートセーブ、入力変更、色テーマ切替を入れること。

# 3. 製品コンセプトとゲーム全体像
本企画の仮題は『Project 継承獣クロニクル』とする。主人公は古びた牧場を継いだ見習い“継承士”。世界各地の『門印』が壊れたことで、各地域の生態と記録が断片化している。主人公は契約した継承獣を育て、6つの門印を修復しながら、記録改ざんを進める組織『黒鐘団』の目的を追う。基本体験は『短く探索する → 契約獣を増やす → 継承合成で系譜をつなぐ → 新地域へ進む』である。

| **ジャンル**       | モンスター育成 + 収集 + 合成を中核にしたトップダウンRPG                                 |
|--------------------|-----------------------------------------------------------------------------------------|
| **プレイ人数**     | 1人。オンライン対戦や交換は初版では非採用。                                             |
| **想定プレイ時間** | メイン30〜35時間、図鑑/希少合成/裏迷宮込みで45〜55時間。                                |
| **対象ユーザー**   | 初代携帯機RPGの感触を求める層、10〜20分単位で遊びたい層、育成計画を立てるのが好きな層。 |
| **世界構造**       | ハブ牧場 + 6地域 + 最終塔 + 裏迷宮。小さな画面単位で区切られたタイル型世界。            |
| **戦闘構造**       | 3体パーティ制ターンバトル。個別入力と戦術AIの切替が可能。                               |
| **固有のフック**   | 継承合成で『見た目は新種、系譜は手持ちの履歴を引き継ぐ』成長履歴を残す。                |

## 3.1 企画の三本柱
| **柱** | **定義**             | **要求される実装**                                                                   |
|--------|----------------------|--------------------------------------------------------------------------------------|
| P1     | 狭い画面でも迷わない | 地形・出口・イベント・NPCの優先度を一画面で整理。探索中の常時HUDは極小。             |
| P2     | 育成履歴が資産になる | 単純な捕獲ではなく、合成と継承で過去の手持ちが次世代の強みになる。                   |
| P3     | 10分でも進む         | 遭遇率・会話量・マップ密度・メニュー手数を調整し、短時間でも前進感がある設計にする。 |

> **非採用 / 初版で切るもの**
>
> オンライン対戦、交換、PvPランク、音声付き会話、オープンワールド、物理演算、手描き高解像度演出、自由視点3D、生成マップ、フルボイス、ガチャ。これらは“初代級の密度”を薄めるか、少人数開発で破綻しやすい。

# 4. コアシステム要件
## 4.1 探索 / マップ / 移動
探索は4方向移動のトップダウン型とし、画面内の可視マス数を20x18で固定する。内部解像度は320x288を標準とし、16x16タイル換算でGame Boyの“20x18画面密度”を継承する。Godotではviewport stretch + integer scalingを採用し、texture filteringはnearest-neighborを前提とする [R5]。これにより“現代ディスプレイ上での読みやすさ”と“携帯機的な密度”を両立できる。

| **ID** | **要件**     | **内容**                                                                          |
|--------|--------------|-----------------------------------------------------------------------------------|
| EX-01  | 内部解像度   | 標準320x288。UI込みで整数拡大する。可視領域は20x18マス。                          |
| EX-02  | タイル単位   | 基本タイル16x16。大型オブジェクトは16x32, 32x32, 32x48を許可。                    |
| EX-03  | 移動         | 4方向のみ。歩行80 px/s、ダッシュ112 px/s。滑りや慣性は持たせない。                |
| EX-04  | 当たり判定   | 1タイル単位衝突を基本にしつつ、Tiled object layerで例外形状を許可。               |
| EX-05  | マップ切替   | 画面切替型ではなく連結スクロール。出入口、ワープ、段差でテンポよく区切る。        |
| EX-06  | 探索情報量   | 1画面あたり主たるインタラクション対象は最大3種。宝箱、NPC、出口が重なり過ぎない。 |
| EX-07  | 遭遇管理     | フィールド/洞窟/遺跡ごとに遭遇テーブルを分離。敵の強さは地域ランクで制御。        |
| EX-08  | オブジェクト | 扉、宝箱、採取点、看板、押しスイッチ、落下床、片道段差、回復泉を標準実装。        |
| EX-09  | 再訪性       | 地域ごとに“今は通れない導線”を最低2箇所配置し、新技能や進行で回収させる。         |

## 4.2 戦闘
戦闘は3体パーティ制のターンバトルとし、携帯機向けに1戦30〜90秒で終わるテンポを狙う。プレイヤーは各体に手入力できる一方、雑魚戦では戦術AI（攻め重視 / 回復優先 / MP節約 / 状態異常重視）へ切替可能とする。これにより、DQM級の育成群管理感と、現代プレイヤーが求める手数削減を両立する。

| **ID** | **要件**   | **内容**                                                             |
|--------|------------|----------------------------------------------------------------------|
| BT-01  | パーティ   | 前衛3体固定。控えは6体まで。戦闘中の交代は1ターン消費。              |
| BT-02  | 基本能力   | HP / ATK / DEF / MAG / RES / SPD / LUK の7値。                       |
| BT-03  | 属性       | 火 / 水 / 風 / 土 / 雷 / 光 / 闇 / 無 の8系統。                      |
| BT-04  | 状態異常   | 毒、火傷、凍結、睡眠、混乱、沈黙、呪印、気絶の8種を標準採用。        |
| BT-05  | 行動       | 通常攻撃、技能、道具、防御、交代、戦術変更、逃走。                   |
| BT-06  | ターン順   | SPD基準の準リアル順番制。乱数補正は±5%以内。                         |
| BT-07  | 雑魚戦時間 | 標準勝利まで2〜4ターン、30〜90秒以内を目標。                         |
| BT-08  | ボス設計   | 単純数値肥大ではなく、護衛、相性、蘇生、行動パターン変化で山を作る。 |
| BT-09  | 倍速       | 1x / 2x / 3x を実装。既読戦闘演出は短縮可能。                        |

> **戦闘の基準式（初版）**
>
> Damage = max(1, Power * (ATK_or_MAG / DEF_or_RES)
> * Element * Crit * Random(0.90, 1.10))
> Heal = Power * (1 + MAG * 0.015)
> Turn = SPD order with small +-5% jitter
> ※ 初版は分かりやすさ優先。式は複雑にし過ぎない。

## 4.3 契約 / 育成 / 継承合成
本企画の中毒性の核は『契約獣を増やす』『育てる』『次世代へ強みを残す』の三層である。野生個体は戦闘後の契約判定、専用餌、リーダー特性、相手残HP、状態異常などで加入率が変動する。合成はLv10以上の2体を消費して卵を作る方式とし、子はレベル1で誕生、系譜・技能・特性を部分継承する。“過去の手持ちが未来の強さになる”ことを、単なる図鑑埋め以上の価値として設計する。

| **ID** | **要件** | **内容**                                                                 |
|--------|----------|--------------------------------------------------------------------------|
| SY-01  | 加入方法 | 通常は戦闘後契約判定。特定種のみイベント加入、卵、特殊合成で獲得。       |
| SY-02  | レベル   | 通常上限50。裏迷宮解放後に70。                                           |
| SY-03  | ランク   | F / E / D / C / B / A / S。地域進行と合成で上位を解放。                  |
| SY-04  | 特性     | 各モンスターは先天2特性、上位種のみ第3特性候補。                         |
| SY-05  | 技能継承 | 親2体から最大3技能を選択継承。重複時は上位版へ統合。                     |
| SY-06  | 特性継承 | 親ごとに1特性候補を持ち込み、子は2つまで装着可能。                       |
| SY-07  | 合成結果 | 基本は family + rank rule で決定。18件の exact recipe で希少種を解放。   |
| SY-08  | 図鑑価値 | 図鑑は捕獲数だけでなく、継承履歴・見つけた配合レシピ・最大到達Lvも記録。 |

> **継承合成の実務的な設計ポイント**
>
> 完全な総当たり配合表を最初から作ると破綻しやすい。初版では『family + rank rule』を主軸にし、物語ややり込みの節目だけに18件程度の exact recipe を置く。これで“配合の発見感”を残しつつ、データ管理量を抑えられる。

# 5. データ要件とマスタ設計
“ちゃんとプレイできる情報量をデータで持つ”ためには、コードより先にマスタの分割単位を固定する必要がある。Tiledはcustom properties、template、worldファイルを持ち [R6]、AsepriteはCLI exportでatlas + jsonを吐ける [R7]。したがって、ゲーム本体は『実行系』、コンテンツは『外部データ』として分離する。デザイナがコードを触らずにモンスター、遭遇、ショップ、NPC、イベント条件を更新できることを必須要件とする。

| **マスタ**       | **最低件数** | **用途**                          | **主担当**     |
|------------------|--------------|-----------------------------------|----------------|
| monster_master   | 120          | 種族・能力・特性・進化/継承元     | ゲームデザイン |
| skill_master     | 84           | 技能効果・消費・対象・属性        | ゲームデザイン |
| trait_master     | 36           | パッシブ特性・条件・重複規則      | ゲームデザイン |
| item_master      | 60           | 回復・触媒・技書・外観アイテム    | ゲームデザイン |
| map_master       | 39           | 各マップの基本情報・BGM・進行条件 | レベルデザイン |

# 7. UI / UX 要件
UIは『探索時は静か、メニュー時は濃い』を基本とする。Godotのpixel art系設定では、viewport + integer scaling が正しい表示例として示され、fractional scaleは線幅やチェッカーが崩れる [R5]。よって全画面を内部解像度基準で作り、動的な高解像度UIを重ねるのではなく、ゲーム本体もUIも同じピクセルグリッドで設計する。

| **画面**       | **主表示**                       | **要件**                                         |
|----------------|----------------------------------|--------------------------------------------------|
| 探索HUD        | エリア名 / ミニ通知              | 常時表示は極小。HP常設はしない。                 |
| メインメニュー | PT / バッグ / 図鑑 / 依頼 / 保存 | 2入力で主要機能へ到達できること。                |
| モンスター詳細 | 能力 / 特性 / 技 / 系譜          | 育成計画に必要な情報を1画面内にまとめる。        |
| 合成画面       | 親2体 / 結果候補 / 継承選択      | 結果プレビューを先に見せる。失敗の恐怖を下げる。 |
| ショップ       | 所持金 / 在庫 / 説明             | 買う前に対象と用途が必ず分かる。                 |
| 戦闘           | 敵 / 味方 / コマンド / 行動順    | メニュー深度を浅く。雑魚戦は2入力以内。          |
| 図鑑           | 発見 / 契約 / 合成 / 系譜        | 進捗が分かる一覧性を優先。                       |
| オプション     | 速度 / ボタン / 色 / 振動        | RETRO風だが現代的に調整できること。              |
| タッチUI       | 仮想D-pad / A / B / Menu         | 位置、サイズ、透過率を変更可能にする。           |

> **探索HUDのレイアウト指針**
>
> [Top-left] エリア名 (短時間表示)
> [Top-right] 一時通知 (入手、図鑑更新、依頼達成)
> [Center] ゲーム画面のみ
> [Bottom] 常設HUDなし
> ※ 情報は画面に常駐させず、行動の邪魔をしない。

> **戦闘UIのレイアウト指針**
>
> [上段] 敵1 / 敵2 / 敵3
> [中段] 効果ログ 1〜2行
> [下段左] 味方3体のHP/MP/状態
> [下段右] こうげき / 技 / 道具 / 防御 / 交代 / 戦術
> ※ 雑魚戦は2入力、ボス戦でも3〜4入力程度で完結する構造にする。

> **アクセシビリティ最小要件**
>
> 文字は8x8基準だが、実表示では十分な整数拡大を行う。戦闘速度、振動、フラッシュ強度、色テーマ、ボタン配置、会話速度、既読スキップ、タッチ透過率を変更可能にする。弱点・耐性は色だけでなくアイコンでも示す。

# 9. AI生成ツール評価と一貫性維持ワークフロー
Google DeepMindのNano Banana prompt guideは、制作向け仕様の明示、複数案の同時生成、参照画像をアップロードしてキャラクターに固有名を与えることで subject consistency を維持できると案内している [R9]。Gemini API/Google blogはNano Banana 2 と Nano Banana Pro の2系統を提示し、Nano Banana 2 を高速・高効率、Nano Banana Pro を高品質用途と位置付けている [R9]。OpenAIの画像生成ガイドは参照画像を使った新規生成、既存画像の編集、マスク編集をサポートし [R8]、GPT Image 1.5 prompting guide は“何を変え、何を変えないかを毎回明示する”差分編集ワークフローを強く推している [R8]。

さらに、Midjourneyは Character Reference と Style Reference を正式提供しており [R10]、Adobe Fireflyは構図参照、スタイル参照、さらに2026年3月時点でStyle IDsによるブランド一貫生成を案内している [R11]。一方、Recraft自身のbest practicesでは dedicated character tracking は持たないと明記している [R12]。したがって、本件で“最終成果物の一貫性”を優先するなら、主戦力は Nano Banana 2/Pro と GPT Image 1.5 の二本立てが妥当である。X上でもGoogleAIStudio公式はNano Banana Proのベストプラクティス共有、GeminiApp公式は16-bit art showcase を出しており、レトロ資産制作の実験が前提機能として扱われ始めている [R14][R15]。

| **ツール**      | **強い点**                                      | **弱い点**                           | **使い所**                           | **判定** |
|-----------------|-------------------------------------------------|--------------------------------------|--------------------------------------|----------|
| Nano Banana 2   | 高速、多案、subject consistency、4K/多AR [R9] | 最終ピクセル精度は人手補正前提       | ムードボード、タイル案、立ち絵差分   | 主力     |
| Nano Banana Pro | 高品質・高忠実度 [R9]                         | 速度/コスト重め                      | 主要キャラ、キービジュアル、最重要案 | 主力     |

## 9.2 一貫性を壊さないための運用ルール
C-01: キャラクターごとに“固定参照パック”を持つ。正面/側面/3/4、色見本、比率メモ、禁止変形をまとめる。

C-02: モデルへ渡す名前を固定する。毎回別名にしない。Nano Banana系は固有名付与でsubject consistencyを上げられる [R9]。

C-03: 1プロンプト1アセット種別。タイル、立ち絵、戦闘顔、UIを同時に作らせない。

C-04: 差分編集では『変える点』『維持点』を毎回再記述する。OpenAI cookbookは invariants 再記述を強く推す [R8]。

C-05: 最終採用アセットは必ず管理表へ記録する。tool / model / date / prompt hash / reference IDs / human cleanup 担当を残す。

C-06: 商用出荷候補のプロンプトには“original artwork only / no logos / no trademarks / no watermarks”を明記する [R8]。

# 10. 技術構成 / 実装方針 / 保存設計
Godot DocsはTileMapLayer/TileSetにより大規模なタイル描画が最適化されると説明しており [R4]、iOS/Android出力も公式にサポートする [R5]。Tiledはworldファイル、template、custom propertiesを備え [R6]、AsepriteはCLI export でatlas + metadataを書き出せる [R7]。この3つを直結した『2DタイルベースRPG向けの王道パイプライン』を構成する。

| **領域**     | **方針**                           | **補足**                                                    |
|--------------|------------------------------------|-------------------------------------------------------------|
| レンダリング | viewport + integer scaling         | fractional scale禁止。テクスチャフィルタはnearest。         |
| マップ       | Tiled JSON読込                     | object layersでNPC/warp/chest/triggerを配置。               |
| アセット     | Atlas化して読み込む                | 読み込みコストとdraw callを抑える。                         |
| 保存         | save slot 3 + autosave 1           | バージョン番号とマイグレーション関数を必須化。              |
| 入力         | ゲームパッド / キーボード / タッチ | すべて同一アクション名へ抽象化。                            |
| 移植         | iOSはmacOS + Xcode前提 [R5]      | リリース直前に慌てないよう、初期から署名/証明書手順を確認。 |

|--------------------|--------------------------------------|--------------------|-----------------------------------------|

| 合成表の爆発       | 総当たり配合を狙う                   | 管理不能           | rule-based を主軸に exact recipe を限定 |
| モバイル可読性不足 | 文字とボタンが小さい                 | 遊べない           | 320x288基準 + 実機検証                  |
| 法務リスク         | 既存IPに寄せ過ぎる                   | 販売不可・差し替え | オリジナル命名・形状・UI・地形          |
| 終盤のバランス崩壊 | 希少配合だけが強過ぎる               | 単調化             | 特性と役割分散でメタを散らす            |

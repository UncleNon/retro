# 2. 調査結果: 何を再現すべきか

> **Archived**: この文書は旧設計。現行の source of truth は `docs/requirements/`。

当時の携帯機RPGを"それっぽく"見せていた本質は、低性能そのものではない。本質は、制約に合わせて情報設計を圧縮し、プレイヤーが一画面内で『今やるべきこと』を迷わないように整理した点にある。Nintendoの仕様ではGame Boyは160 x 144画素・4階調 [R1]、Pan Docsでは8x8/8x16 sprite、40 sprite/画面・10 sprite/lineが示されている [R2]。このため、当時の優れた作品は背景・UI・キャラを同時に主張させず、"一目で判別できる差分"だけを残した。ここを再現しない限り、単にドット化しても初代級の品質にはならない。

| **観測された事実**                                  | **画面上の結果**                         | **本企画での要件化**                                                       |
|-----------------------------------------------------|------------------------------------------|----------------------------------------------------------------------------|
| 160x144 / 20x18タイル級の狭い視界 [R1][R2]         | 常時表示情報が少ない。探索が密に感じる。 | 可視20x18マスを維持。常時HUDは最小化し、メニューに機能を寄せる。           |
| スプライト数/線数制限 [R2]                          | 同一画面の被写体数が抑制される。         | 街・ルート・ダンジョンごとに同時注目対象を3つ以内に制御する。              |
| 4階調や少色数 [R1][R2]                              | 輪郭と塊の読みやすさが最優先になる。     | 1キャラ1特徴、1タイル1役割、1UI面1主題で設計する。                         |
| 現代スマホでのRETRO移植成立 [R3]                    | 現代端末でも成立する。                   | モバイル操作は追加するが、ゲームループ自体は古典的でよい。                 |
| UnityのPixel Perfect Camera [R5]                    | 中途半端な拡大でドットが滲まない。       | 内部解像度固定 + 整数拡大を標準とする。                                    |

## 2.1 再現原則

原則A: 1画面1課題。プレイヤーが『話す』『拾う』『進む』『戻る』のどれをすべきか一目で分かること。

原則B: 1キャラ1記号。髪型・角・尻尾・帽子・肩幅など、遠目でも識別できる差異を優先すること。

原則C: 地形は文法で作る。草・土・崖・水・壁の接続ルールを固定し、バリエーションは文法の範囲で増やすこと。

原則D: UIは常時主張しない。探索時は静か、戦闘とメニューでのみ情報密度を上げること。

原則E: 収集→強化→新領域解放の1サイクルを10〜20分で回せるようにすること。

原則F: ドットの精度は"解像度の低さ"ではなく"塊の整理"で決まる。ぼかしやAAに逃げないこと。

原則G: データは全部外に出す。モンスター、技能、遭遇、ショップ、イベント条件はコードに焼き込まないこと。

原則H: "昔っぽい"ではなく"今遊んでも気持ちいい"。倍速、オートセーブ、入力変更、色テーマ切替を入れること。
